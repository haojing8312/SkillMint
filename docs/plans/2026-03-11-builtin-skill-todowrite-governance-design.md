# Builtin Skill TodoWrite Governance Design

**背景**

当前聊天与任务面板已经逐步收敛到“右侧面板承担当前任务状态”的方向，但 builtin skill 侧仍存在一个根问题：

- 某些 builtin skill 会先进入多步骤执行流程
- 但在真正写入 `TodoWrite` 之前，UI 已经需要表达“当前任务”
- 于是前端容易补充自己的推断状态，导致状态来源分裂

“智能体员工助手”就是典型案例。用户补充信息、生成草案、等待确认、执行创建，本质上都是同一条计划中的步骤，但当前 skill 文本并没有强制要求在首轮建立 `TodoWrite` 计划，因此 UI 与 skill 生命周期之间出现断层。

## 目标

1. 所有“多步骤执行”的 builtin skill 使用统一任务状态来源。
2. `TodoWrite` 成为这类 skill 的主任务状态源，不再由前端猜测业务阶段。
3. “收集信息”“等待确认”被纳入正式计划步骤，而不是计划外阶段。
4. skill 作者有统一规则可遵循，避免每个 skill 各自定义任务生命周期。

## 非目标

- 本次不要求所有普通问答型 skill 都使用 `TodoWrite`。
- 本次不立即上运行时硬拦截机制。
- 本次不重做全部聊天 UI，只统一状态治理规则。

## 方案对比

### 方案 A：仅改各 builtin skill 文本

做法：

- 在每个 builtin skill 的 `SKILL.md` 中补充“尽早使用 TodoWrite”

优点：

- 改动面小

缺点：

- 约束软，容易漏掉
- 不形成统一平台规范

### 方案 B：建立统一平台规范 + skill 层落地

做法：

- 在系统提示词中定义 builtin skill 的统一规则
- 在各 builtin skill 的 `SKILL.md` 中写出本领域计划模板

优点：

- 统一性最好
- 前端只需要消费同一套任务源
- 保持较低实现复杂度

缺点：

- 需要先定义“什么算多步骤执行”

### 方案 C：运行时硬校验

做法：

- 对多步骤 builtin skill 首轮是否写入 `TodoWrite` 进行运行时拦截或降级

优点：

- 最强约束

缺点：

- 实现复杂
- 误判成本高
- 当前阶段过重

## 结论

采用方案 B。

即：

- 统一规则写进系统提示词
- builtin skill 自身写出明确计划模板
- UI 继续只认 `TodoWrite` 和真实运行态

## 多步骤 builtin skill 的定义

满足任一条，即认定为“多步骤执行 builtin skill”，必须使用 `TodoWrite`：

1. 需要先收集用户信息再执行。
2. 需要先给草案/方案，再等待用户确认。
3. 会创建、修改、删除、发布、安装、配置某个对象。
4. 需要跨多个工具或多个阶段产出结果。
5. 用户期望的是“流程交付”而不是单轮问答。

不满足以上条件的简单问答型 builtin skill，可不强制 `TodoWrite`。

## 统一规则

### 规则 1：首轮建立计划

对于多步骤 builtin skill，在首轮可执行响应前必须调用 `TodoWrite`，建立完整计划。

这里的“首轮可执行响应前”指：

- 在开始连续执行流程之前
- 在展示阶段性工作状态之前
- 在 ask_user 或执行动作之前，至少先建立计划骨架

### 规则 2：收集信息属于正式步骤

收集信息不能被视为计划外动作，必须作为计划中的一个或多个正式步骤。

例如：

- 收集关键需求
- 确认目标对象
- 补齐必要配置

### 规则 3：等待确认属于正式步骤

所有需要用户确认后才能执行的 builtin skill，必须把“等待用户确认”写成计划中的正式步骤。

### 规则 4：执行步骤不能提前进入进行中

在用户尚未确认前，不得把会产生副作用的执行步骤标记为 `in_progress`。

例如：

- 创建员工
- 修改员工
- 安装技能
- 发布配置

在确认前，这些步骤应保持 `pending`。

### 规则 5：计划状态随交互推进

交互推进顺序应遵循：

1. 建立计划
2. 收集信息步骤 `in_progress`
3. 草案步骤 `completed`
4. 确认步骤 `in_progress`
5. 执行步骤 `in_progress`
6. 结果步骤 `completed`

## 前端状态原则

前端不再自行推断“业务阶段”。

前端只消费：

1. `TodoWrite` 计划状态
2. 真实运行态事件
   - 运行中的工具
   - `ask_user`
   - 风险确认/执行确认

因此：

- “等待收集需求”应来自计划第一步，而不是 UI 额外生成
- “等待确认”应来自计划中的确认步骤，而不是 UI 猜测

## Builtin Skill 编写模板

所有多步骤 builtin skill 的 `SKILL.md` 应显式包含以下要求：

1. 在首轮建立计划
2. 列出建议计划步骤
3. 说明哪些步骤在确认前不能执行
4. 要求在阶段切换时更新 `TodoWrite`

示例模板：

```md
在开始执行前，先使用 TodoWrite 建立任务计划。计划至少包括：
1. 收集必要信息
2. 盘点已有资源
3. 生成草案
4. 等待用户确认
5. 执行变更
6. 返回结果与建议

在用户确认前，不得执行会产生副作用的步骤。
每当阶段推进时，都要更新 TodoWrite 状态。
```

## 员工助手示例

“智能体员工助手”应改为在最开始写入类似计划：

1. 收集关键需求信息
2. 盘点已有员工与技能
3. 生成配置草案
4. 等待用户确认
5. 执行创建或修改
6. 返回结果与后续建议

这样：

- 用户补充信息属于步骤 1
- 草案生成属于步骤 3
- “确认创建 / 确认修改”属于步骤 4
- 真正落库属于步骤 5

## 实施策略

### 第一阶段

- 修改系统提示词，加入统一治理规则
- 修改重点 builtin skill（先从员工助手开始）以遵守新规范

### 第二阶段

- 扫描所有 builtin skill，补齐计划模板
- 为典型 skill 增加测试用例

### 第三阶段

- 视效果决定是否增加运行时告警或校验

## 测试分层

这类治理规则优先放在轻量层验证，而不是直接放进重型 Tauri 集成测试。

推荐分层：

1. `runtime-skill-core`
- 验证多步骤 builtin skill 的分类规则
- 验证治理规则文本注入
- 验证 builtin skill 文本本身是否包含 `todo_write` 约束

2. `src-tauri`
- 只验证 prompt 装配链路是否消费了轻量层提供的规则
- 避免为纯字符串规则新增长耗时的重型集成测试

3. 前端
- 继续验证 UI 只消费 `TodoWrite` 和真实运行态，不自行发明业务阶段状态

这样可以避免每次为了验证一条 prompt 规则，就重新编译整条 Tauri runtime 依赖链。

## 验收标准

1. 多步骤 builtin skill 在首轮就写入 `TodoWrite`。
2. “收集信息”与“等待确认”都能在任务清单中体现。
3. 前端不再需要为这类 skill 发明业务阶段状态。
4. builtin skill 文本具备统一编排规范。
5. “智能体员工助手”能完整覆盖需求收集、草案确认、执行与交付全过程。
