# WorkClaw 3-7天最小商业化验证闭环

任务：`t_783e862f`

日期：2026-05-12

## 0. 任务边界

### 目标

为 WorkClaw 新手友好的桌面 AI 智能体 / AI 团队指挥产品，设计一个 3-7 天内可执行、可收集反馈、可反向约束 MVP 范围的最小商业化验证闭环。

### 输入

- 当前产品方向：Hermes-aligned self-improving profile runtime，`profile_id -> AI employee runtime home`。
- 当前工程约束：不把 OpenClaw 作为新兼容目标，不新增 sidecar 依赖；OpenClaw 只作为历史/迁移输入。
- 当前协作方式：WorkClaw 项目通过 Hermes Kanban task_id 推进，飞书群只做进度可见层。
- 已有用户手册线索：开始任务、智能体员工中心、专家技能中心、飞书接待、长期记忆管理。

### 输出

1. 1-2 个优先试点客户画像 / 场景假设。
2. 一个最小商业化闭环：触达话术、演示场景、试用成功标准、反馈收集方式。
3. 3-5 个本周可验证商业指标。
4. 对产品 MVP 范围的商业化约束输入。

### 风险与安全边界

- 本方案不包含对外价格、合同、交付周期承诺。
- 任何对外报价、签约、客户交付时间、数据安全/隐私承诺，均为高风险事项，必须由郝敬最终确认。
- 试点任务只选择低敏、可脱敏、非生产关键任务；不得要求试点客户提供密钥、token、商业秘密或生产系统权限。

## 1. 优先试点客户画像 / 场景假设

### ICP A：AI-native 超级个体 / 小团队负责人（优先级 P0）

| 维度 | 描述 |
| --- | --- |
| 典型身份 | 独立开发者、AI 咨询顾问、产品/运营负责人、内容/知识产品创业者、1-5 人轻团队负责人 |
| 核心痛点 | 已经会用 ChatGPT/Claude/Cursor，但任务仍散落在聊天窗口、文档和群消息里；缺少“AI 员工身份 + 任务进度 + 结果沉淀 + 可复用技能”的闭环 |
| 购买/试点动机 | 想把个人工作流升级成“可指挥的 AI 团队”，减少重复交代上下文，提高跨任务复用与可追踪性 |
| 适配原因 | WorkClaw 的桌面入口、员工中心、技能中心、长期记忆、Kanban/飞书进度通报，能直接映射到他们的日常工作流 |
| 试点场景假设 | 用户给出一个真实但低敏任务，例如“为我的 AI 产品设计 7 天验证计划”，WorkClaw 通过 2-4 个专家员工拆解、执行、沉淀结果，并让用户看到进度与最终方案 |
| 不适配信号 | 只想要单轮聊天；完全不接受桌面端；需要企业级权限/审计/私有化/合同流程才能试用 |

### ICP B：AI 项目交付型团队 / 解决方案工作室（优先级 P1）

| 维度 | 描述 |
| --- | --- |
| 典型身份 | AI 应用开发工作室、自动化顾问、企业内创新小组、需要并行推动多个客户 PoC 的项目负责人 |
| 核心痛点 | 多 Agent / 多角色协作容易失控：任务没有稳定 task_id，过程只在群聊里，交付物难追踪，复盘难沉淀 |
| 购买/试点动机 | 想把 AI 项目执行从“人肉群聊协调”升级为“Kanban 状态机 + 专家 profile + 结果回写 + 飞书可见进度” |
| 适配原因 | WorkClaw 当前 Kanban 协作试点就是可展示样板：PM 建卡、专家认领、开始/阻塞/完成通报、summary/metadata 回写 |
| 试点场景假设 | 用 WorkClaw 跑一个内部客户方案任务，例如“拆解一个 3 天 AI 自动化 PoC 方案”，展示 task_id、专家分工、进度通报、结果沉淀 |
| 不适配信号 | 要求立即接入客户生产环境；要求先签企业合同/定制开发；强依赖未实现的权限、审计、私有化能力 |

### 本周优先选择建议

优先选择 ICP A 的 2-3 位熟人或弱关系线索做快速验证；若其中有交付型团队负责人，再顺带验证 ICP B。原因：ICP A 的试用门槛最低、决策链最短、反馈速度最快，更适合 3-7 天闭环。

## 3. 最小商业化验证闭环

### 3.1 闭环总览

```text
候选线索筛选
  -> 轻量触达
  -> 资格确认
  -> 20-30分钟桌面演示
  -> 3-7天低敏试用/陪跑
  -> 结构化反馈
  -> 复用意愿/下一步判断
  -> 回写 Kanban 与产品约束
```

### 3.2 触达话术

#### 熟人/弱关系首条触达

```text
我正在验证 WorkClaw：一个面向个人和小团队的桌面 AI 智能体 / AI 团队指挥产品。

它想解决的不是“再做一个聊天窗口”，而是让你能创建/选择 AI 员工，把一个真实任务交给多个专家角色协作推进，并看到 task_id、进度、结果沉淀和可复用能力。

我这周只找少量试点用户做 20-30 分钟演示，不涉及报价/合同/交付承诺。想请你用一个低敏真实任务帮我判断：这个方向是否值得继续做、哪一步最有价值、哪里会卡住。

如果你愿意，我先发你 3 个确认问题，合适的话约一次演示。
```

#### 资格确认 3 问

```text
1. 你最近有没有一个反复要交代上下文、需要多个 AI/角色协作的任务？
2. 这个任务能不能脱敏，避免提供密钥、客户隐私或生产系统权限？
3. 你是否愿意在演示后给 10 分钟反馈：哪里有用、哪里看不懂、是否愿意再用一次？
```

#### 演示邀请

```text
我建议演示用你的一个低敏任务现场跑：
- 先在桌面端选择/创建一个 AI 员工；
- 再把任务交给 WorkClaw 的专家/团队协作流；
- 中间看 task_id、进度、结果沉淀；
- 最后一起判断是否能替代你现在的“聊天窗口 + 文档 + 群消息”流程。

说明：这次只做产品验证，不做价格、合同或交付周期承诺；这些需要郝敬单独确认。
```

#### 演示后跟进

```text
谢谢你刚才看 WorkClaw。为了判断这是不是一个值得继续推进的产品，我想记录 4 个点：
1. 你最想复用的一个能力是什么？
2. 哪一步让你困惑或不信任？
3. 如果本周再给你一次低敏任务，你是否愿意继续用？为什么？
4. 如果不愿意继续用，最大阻塞是产品能力、使用成本、信任、安全，还是场景不匹配？
```

## 4. 桌面演示场景

### 推荐演示主题

优先使用“用户自己的低敏真实任务”。如果用户没有现成任务，使用默认演示任务：

```text
请帮我为一个 AI 工具产品设计 7 天最小商业化验证计划：包括目标客户、触达话术、演示流程、成功指标和下一步任务拆解。
```

### 20-30 分钟演示流程

| 阶段 | 时间 | 操作 | 需要在桌面端可见的能力 |
| --- | --- | --- | --- |
| 1. 定位痛点 | 3 分钟 | 说明当前痛点：聊天窗口无法承载 AI 团队协作、进度和沉淀 | 产品定位页/开始任务入口清晰 |
| 2. 选择/创建 AI 员工 | 5 分钟 | 展示智能体员工中心，选择一个已有员工；如条件允许，展示员工助手创建/调整员工 | 员工列表、员工详情、开始任务入口、Profile Home/长期记忆状态条至少可解释 |
| 3. 发起任务 | 5 分钟 | 输入一条业务任务，明确目标、限制、输出格式 | 开始任务页、输入框、会话创建、首条回复 |
| 4. 团队/专家协作 | 7 分钟 | 展示任务被拆解/流转的样板：task_id、专家角色、开始/完成进度、结果回写 | Kanban task_id、状态、summary/metadata；若飞书通报已配置，展示群内开始/完成消息 |
| 5. 结果沉淀 | 5 分钟 | 展示结果在会话历史、记忆/技能/文档中的沉淀方式 | 会话历史、长期记忆/项目记忆说明、技能中心或输出文档路径 |
| 6. 反馈收集 | 5 分钟 | 当场记录 4 个反馈问题和是否愿意再次试用 | 反馈表/飞书记录/Kanban comment 均可；必须能追溯到 task_id |

### 演示必须可见能力清单

P0 必须可见：

1. 桌面端能从“开始任务”发起一次任务，并返回结果。
2. 用户能看到 AI 员工 / profile 的存在，而不是只有一个泛聊天机器人。
3. 至少一个任务有稳定 task_id 或等价进度记录，能说明“任务不是散落在聊天里”。
4. 结果能被沉淀到会话历史、文档或 Kanban summary，方便复盘。
5. 风险边界清楚：不索要敏感凭据，不承诺生产接入，不承诺外部交付周期。

P1 尽量可见：

1. 员工助手创建/调整员工。
2. 长期记忆 / Profile Home 状态。
3. 专家技能中心：技能查看、开始任务或安装入口。
4. 飞书/Kanban 进度通报：开始、阻塞、完成三类消息。
5. 多专家协作的任务图或 PM 汇总样板。

本周不建议作为演示主卖点：

1. 价格、合同、付款、企业采购流程。
2. 私有化部署、企业级权限审计、复杂组织管理。
3. 大规模技能市场/商业分发。
4. OpenClaw 兼容能力；如被问到，只说明它是历史/迁移输入，不是新产品承诺。

## 5. 3-7天试用成功标准

### 用户级成功标准

一次试用视为“有效成功”，需要满足以下 4 条中的至少 3 条：

1. 用户用自己的低敏任务完成一次从发起到结果的闭环。
2. 用户能说清 WorkClaw 相比单个聊天窗口多解决了什么问题。
3. 用户愿意在 3-7 天内再用一次或推荐给一个相似用户试看。
4. 用户给出一个明确的阻塞/改进点，可转成产品或工程 Kanban 任务。

### 产品级成功标准

本周验证结束时，至少沉淀：

1. 每位试点用户的 ICP 标签、触达来源、演示任务、task_id、最终结果路径。
2. 每次演示的通过/失败原因。
3. 至少 3 条可转成产品/工程/客户成功任务的反馈。
4. 对 MVP must-have / should-have / won't-do 的更新建议。

## 7. 本周可验证商业指标

| 指标 | 定义 | 采集方式 | 3-7天判断口径 |
| --- | --- | --- | --- |
| 试点意向率 | 有效触达后，愿意回答资格确认问题或看演示的人数 / 有效触达人数 | 手工表格或飞书记录 | 验证价值主张是否能让目标人群愿意给时间 |
| 演示转试用率 | 完成演示后，愿意用一个低敏真实任务继续试用的人数 / 完成演示人数 | 演示记录 + 跟进消息 | 验证“桌面 AI 团队指挥”是否比普通聊天更有吸引力 |
| 首任务完成率 | 试用中完成一次发起 -> 进度 -> 结果沉淀闭环的人数 / 开始试用人数 | task_id、会话历史、文档路径 | 验证产品最小可用性 |
| 复用意愿率 | 明确愿意本周再用一次或推荐给相似用户的人数 / 有效试用人数 | 反馈表问题 3 | 验证是否有持续价值，而非一次性好奇 |
| 可转任务密度 | 每次有效试用产出的可执行 Kanban 任务数 | 反馈表 + PM 整理 | 验证反馈是否足够具体，能驱动产品迭代 |

说明：以上指标均可人工或半自动收集，不依赖埋点系统，不构成对外商业承诺。

## 8. 产品 MVP 范围的商业化约束输入

### Must-have：没有这些，不建议做商业化演示

1. 新手能在桌面端找到“开始任务”，并完成一次首任务。
2. AI 员工 / profile 概念必须可见：用户能理解“我在指挥某个员工/专家”，而不是只在用聊天框。
3. 任务进度必须可追踪：至少有 task_id、状态、summary/metadata 或等价记录。
4. 结果必须可沉淀：会话历史、文档、记忆或 Kanban summary 至少有一个稳定落点。
5. 试点过程必须能导出/复制反馈记录，方便 PM 汇总。
6. 产品说明必须清楚标记：不索要敏感凭据、不直接接入生产系统、不承诺外部交付周期。

### Should-have：有助于提升演示转化

1. 员工助手能用自然语言创建/调整员工，并回到员工列表看到结果。
2. Profile Home / 长期记忆状态在员工详情中可解释，不需要用户理解底层文件结构。
3. 专家技能中心能展示“技能让输出可复用”，至少能从技能卡片进入任务。
4. 飞书/Kanban 通报能作为“AI 团队协作可视化”样板展示。
5. 演示任务结束后能一键复制“试点记录模板”或生成 PM 汇总。

### Won't-do：本周商业化验证不需要

1. 不做付款、报价、合同、套餐页。
2. 不做企业级 RBAC、审计报表、私有化部署承诺。
3. 不把 OpenClaw 兼容作为新卖点。
4. 不承诺对客户生产环境做自动化操作。
5. 不追求大规模线索投放；只做 2-5 个高质量试点反馈。

### 高风险事项：必须由郝敬确认

1. 对外价格、折扣、合同、试用期条款。
2. 对外交付周期、客户定制范围、生产环境接入。
3. 客户数据、隐私、权限、安全承诺。
4. 是否把试点反馈或客户名称用于公开案例。
