# Feishu Guided Onboarding Design

**Date:** 2026-03-21

## Goal
把飞书首次接入从“配置页里自己理解依赖关系”改成“可跳过的引导式流程”，让新手用户能在最少认知负担下完成接入；同时保留现有设置页作为状态查看、按需修改和排障入口。

## Problem Summary
- 现有飞书设置页同时承担首次接入、参数修改、故障排查三种职责，信息量过大。
- “绑定已有机器人”和“完成飞书授权”被并列展示，但实际上存在依赖关系，容易误导用户。
- “新建机器人”路径和“绑定已有机器人”路径被混在一起，导致按钮前置条件冲突。
- 新手用户不知道当前卡在哪一步，也不知道下一步该做什么。
- 页面内部分按钮在失败时只显示弱提示，体感像“没反应”。

## Product Principles
- 首次接入用向导，不用控制台式设置页。
- 飞书接入是可选能力，在快速设置里必须允许跳过。
- 每一步只让用户做一个决定。
- 当前不能继续时，必须明确说明原因、影响和下一步。
- 向导负责“接通”，设置页负责“修改和排障”。

## Recommended Information Architecture

### 1. First-use surface
在现有“快速设置”体系中增加一个可跳过的“飞书接入”卡片。

用户可以：
- 立即开始配置飞书
- 暂时跳过，先进入产品主界面

跳过后：
- 不阻塞主流程
- 在需要飞书能力的位置显示“尚未完成飞书接入”
- 可从渠道连接器页重新打开向导

### 2. Ongoing surface
“渠道连接器 > 飞书”页改为“控制台”角色，顶部增加“重新打开飞书接入向导”入口。

设置页仅负责：
- 当前状态总览
- 重新打开向导
- 高级修改
- 诊断和日志

## Guided Onboarding Flow

### Step 1: Check environment
目的：确认当前电脑是否具备运行飞书官方插件的条件。

显示内容：
- Node.js 状态
- npm 状态
- WorkClaw 内置运行脚本状态
- 当前检查结论

按钮：
- 重新检测
- 下一步
- 暂时跳过

阻塞规则：
- 环境不满足时不能继续到安装插件

### Step 2: Install official plugin
目的：安装飞书官方插件。

显示内容：
- 插件是否已安装
- 插件版本
- 最近安装错误

按钮：
- 安装插件
- 重试安装
- 下一步

阻塞规则：
- 插件未安装时不能继续到接入方式选择

### Step 3: Choose onboarding path
目的：明确用户到底是“已有机器人”还是“需要新建机器人”。

两个主选项：
- 绑定已有机器人
- 新建机器人

设计要求：
- 用两张大卡片而不是堆在同一页的多个表单
- 给出“适合谁”的一句说明

### Step 4A: Bind existing robot
适用于已有 App ID / App Secret 的用户。

显示内容：
- App ID
- App Secret
- 机器人校验结果

按钮：
- 验证机器人信息
- 保存并继续

阻塞规则：
- 未填写或未通过校验时，不允许进入授权步骤

### Step 4B: Create new robot
适用于没有机器人信息的用户。

显示内容：
- 向导当前状态
- 当前提示词
- 最近输出

按钮：
- 启动新建机器人向导
- 发送输入
- 终止向导
- 读取并继续

设计要求：
- 不要求用户预先准备 App ID / App Secret
- 新建完成后，应尽量自动回填 App ID / App Secret；做不到时至少明确告诉用户去哪里复制

### Step 5: Start runtime and finish authorization
目的：真正建立飞书连接并完成授权。

显示内容：
- 插件安装状态
- runtime 运行状态
- 授权状态
- 最近问题

按钮：
- 启动并授权
- 刷新授权状态

阻塞规则：
- 只有在已有机器人信息或已完成新建机器人后，才能执行启动
- 只有授权通过后，才能进入接待设置

### Step 6: Configure reception
目的：让飞书消息真正进入正确的员工接待链路。

显示内容：
- 默认接待员工状态
- 群聊范围规则数量
- 当前是否已具备接待条件

按钮：
- 去配置接待员工
- 稍后再配
- 完成

规则：
- 这一步不一定要阻塞整个向导完成，但必须明确提示“未配置接待员工时，消息不会稳定分配”

## Quick Setup Integration
- 快速设置中把飞书作为一个独立可选项。
- 如果用户跳过，记录本地状态，后续不再强打断。
- 若用户后续首次进入飞书设置页，顶部默认展示“尚未完成接入，点击继续向导”。

## Feishu Settings Console Structure

### A. Status overview
用一屏卡片展示：
- 环境
- 插件
- 机器人信息
- 授权
- 接待状态

### B. Re-open wizard
明显的主按钮：
- 继续接入向导
- 重新开始接入向导

### C. Advanced changes
保留原来的：
- 机器人信息修改
- 安装/启动控制
- 授权刷新
- 高级配置

### D. Diagnostics
保留：
- 连接摘要
- 最近问题
- 原始日志
- 复制诊断信息

## Error Handling Requirements
- 任何阻塞操作都必须有明显的 loading 状态。
- 任何失败都必须在当前步骤内展示原因，而不是只在页面顶部显示弱提示。
- 用户要能看出“失败在安装”“失败在启动”“失败在授权”“失败在接待配置”中的哪一步。
- 对于 npm、Node、脚本缺失等环境类问题，错误文案要可执行，不能只给技术栈名词。

## State Model
建议把飞书接入进度抽象成单独的 onboarding state，而不是完全依赖设置页临时推导。

最少应包含：
- environment_ready
- plugin_installed
- onboarding_path: `existing` | `create` | `unknown`
- credentials_ready
- runtime_started
- authorization_status
- routing_ready
- skipped
- last_error

## Success Criteria
- 新手用户第一次进入飞书接入时，不需要理解所有内部概念也能完成流程。
- “新建机器人”路径不再要求预先填写 App ID / App Secret。
- “绑定已有机器人”路径和“新建机器人”路径的前置条件清晰分离。
- 用户在任何一步失败时，能明确知道失败原因和下一步。
- 设置页不再承担首次接入教学职责。

## Non-goals
- 当前不改飞书底层协议。
- 当前不更换官方插件主路径。
- 当前不引入新的外部依赖或新的飞书接入模式。
