# 多智能体员工协作验收清单（2026-03-05）

## 1. 验收目标
- 验证飞书群完整闭环：`主员工接管 -> 子员工执行 -> 主员工汇总`
- 验证双端一致：桌面端与飞书端均可感知过程态（流式、委派、澄清、失败兜底）
- 验证用户体验：无需理解内部“线程绑定”等概念即可完成任务

## 2. 前置条件
- 已创建 2 个员工并启用：
  - 主员工：`项目经理`（`is_default = true`）
  - 子员工：`开发团队`
- 2 个飞书机器人已分别配置到对应员工，并拉入同一飞书群
- 桌面应用正在运行，并保持在“开始任务/会话列表”可见状态

## 4. 手工验收脚本（按顺序执行）
> 可直接复制的飞书消息模板见：`docs/plans/2026-03-05-feishu-manual-test-prompts.md`

1. 主员工自动接管（无 @）
   - 飞书发送：`请你们帮我分析一个需求并给出方案和报价。`
   - 桌面预期：
     - 新建/命中飞书会话
     - 聊天头部有“飞书同步”来源标记
     - 协作状态条出现主员工处理态
   - 飞书预期：
     - 收到主员工分段回复（含角色前缀，如 `[项目经理] ...`）

2. 主员工委派子员工
   - 飞书发送：`请细化技术方案，给出架构拆分。`
   - 桌面预期：
     - 出现委派卡片：`项目经理 已将任务分配给 开发团队`
     - 状态文案经历：`已委派` -> `正在汇总最终答复` -> `已输出最终汇总`
   - 飞书预期：
     - 可看到子员工阶段输出（`[开发团队] ...`）
     - 最后有主员工汇总（`[项目经理] ...`）

3. 子员工直达（显式 @）
   - 飞书发送：`@开发团队 你细化一下技术方案`
   - 桌面预期：
     - 本条任务直达开发团队处理，不绕回主员工
     - 协作时间线出现“子员工”事件
   - 飞书预期：
     - 优先看到 `[开发团队]` 回复

4. 文本 @ 兜底（role_id 缺失场景）
   - 飞书发送：`@开发团队 请继续拆分模块`
   - 条件：消息事件中即使没有结构化 `role_id`，仅文本含 `@开发团队`
   - 预期：
     - 仍命中开发团队，不回退主员工
     - 桌面/飞书双方都看到开发团队执行结果

5. 澄清请求闭环
   - 触发方式：让任务进入需要用户确认（ask-user）的步骤
   - 飞书预期：
     - 收到澄清问题（包含选项）
     - 用户直接回复后，流程继续
   - 桌面预期：
     - 出现“需要你的确认”卡片
     - 用户回复后等待态结束并继续执行

6. 失败兜底（发送失败重试）
   - 操作：临时制造飞书发送失败（网络或凭证问题）
   - 预期：
     - 系统自动重试
     - 超过上限后降级为“仅桌面继续”，主流程不中断

7. 会话可达与返回首页
   - 操作：
     - 选中飞书同步会话
     - 点击左侧“开始任务”
   - 预期：
     - 可返回首页，不需要删除会话

## 5. 常见失败排查
- 现象：飞书有消息，桌面没新会话
  - 检查员工是否 `enabled=true`
  - 检查主员工是否仍为 `is_default=true`
  - 检查飞书机器人是否在群内且连接状态正常
- 现象：`@开发团队` 仍落到主员工
  - 检查员工名称/employee_id/role_id 与 @ 文本是否可匹配
  - 检查是否误用了不可见字符或空格分隔异常
- 现象：飞书不回包但桌面继续
  - 属于发送降级路径，优先检查 sidecar 与飞书凭证

- 手工脚本 7 项均通过
