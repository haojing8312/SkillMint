# 新增模型自动切换默认设计

## 1. 目标

解决“模型连接”中新增第二个模型后仍然沿用第一个默认模型的问题。

本次目标是：

1. 新增模型保存成功后，自动切换为默认模型
2. 该行为只对“新增模型”生效
3. 编辑已有模型时，不自动抢占默认
4. 模型列表提供显式“设为默认”入口
5. 任意时刻只允许一个默认模型

## 2. 当前问题

当前实现中，模型默认逻辑存在两个缺口：

1. 前端新增模型时，仅在模型列表为空时才将 `is_default` 设为 `true`
2. 后端缺少模型版 `set_default_*` 命令，无法像搜索引擎那样显式切换默认

因此当前行为是：

1. 第一个模型自动成为默认
2. 第二个及后续模型默认都不是默认
3. 用户若想让第二个模型生效，只能删除第一个默认模型或依赖非直观路径

这与用户预期“刚新配好的模型应立即生效”不一致。

## 3. 范围与边界

本次只处理模型连接默认切换，不涉及搜索引擎、翻译模型、团队路由规则等其他默认逻辑。

明确边界如下：

1. 新增模型：保存成功后自动切换为默认
2. 编辑模型：保持原默认状态不变
3. 手动设为默认：允许
4. 删除默认模型：自动补一个新的默认模型
5. 删除非默认模型：不影响当前默认

## 4. 方案选择

### 方案 A：仅修改新增保存逻辑

做法：

1. 新增模型时，前端直接传 `is_default = true`
2. 后端保存时同时清理其他模型默认标记

优点：

1. 改动最小

缺点：

1. 默认切换逻辑混在保存逻辑里，职责不清
2. 后续手动切默认仍需额外补能力
3. 不利于和搜索引擎默认行为保持一致

### 方案 B：新增独立 `set_default_model` 命令

做法：

1. `save_model_config` 负责保存模型
2. `set_default_model(model_id)` 负责互斥切换默认
3. 前端在新增成功后调用 `set_default_model`

优点：

1. 职责清晰
2. 和搜索引擎默认逻辑一致
3. 支持新增自动切换与手动设为默认两类场景

缺点：

1. 需要让保存接口返回模型 id，或补额外查询逻辑

### 方案 C：前端本地排序推断默认，不做后端约束

优点：

1. 改动看似少

缺点：

1. 数据层可能出现多个默认或没有默认
2. 前后端状态容易不一致
3. 不可接受

### 结论

采用方案 B：新增独立 `set_default_model(model_id)` 命令，由后端保证默认模型唯一性。

## 5. 交互设计

### 5.1 新增模型

用户在“模型连接”中新增模型并点击保存后：

1. 模型保存成功
2. 新模型自动切换为默认
3. 列表中“默认”标签移动到新模型
4. 原默认模型自动取消默认

该行为不额外弹确认框。

原因：

1. 这是用户明确的产品预期
2. 新增模型通常意味着希望立即启用该连接
3. 额外确认会增加接入阻力

### 5.2 编辑模型

编辑已有模型时：

1. 保存仅更新名称、Base URL、模型名、API Key 等字段
2. 不自动修改默认状态

原因：

1. 编辑更多是维护动作，不应意外切换路由
2. 可避免“只是改个 Key 却切了默认”的误操作

### 5.3 模型列表

模型列表行为调整为：

1. 默认模型显示 `默认` 标签
2. 非默认模型显示 `设为默认` 按钮
3. 保留 `编辑`、`删除` 按钮

这样新增自动切换与手动切换两条路径都清晰可见。

## 6. 数据与命令设计

### 6.1 保存模型

现有 `save_model_config` 仍保留，但建议改为返回保存后的模型 id：

- 当前：`Result<(), String>`
- 调整后：`Result<String, String>`

原因：

1. 新增成功后前端需要拿到真实 id 才能调用 `set_default_model`
2. 避免前端通过名称等弱条件反查

### 6.2 设置默认模型

新增命令：

- `set_default_model(model_id: String)`

逻辑：

1. 清除所有非搜索模型的 `is_default`
2. 将目标模型 `is_default` 设为 `1`

SQL 语义应与 `set_default_search` 保持一致，只是筛选条件改为非 `search_%` 模型。

### 6.3 删除默认模型后的补默认

删除模型时需要补全默认生命周期：

1. 查询被删模型是否为默认
2. 执行删除
3. 若被删模型为默认，且仍有剩余模型
4. 选择一个剩余模型补为默认，推荐取列表第一个或最近保存项

本次建议优先采用“剩余列表第一个”策略，简单稳定，不引入额外排序字段设计。

## 7. 前端流程设计

### 7.1 SettingsView 新增模型流程

当前新增保存逻辑：

1. 调用 `save_model_config`
2. 新增时仅在 `models.length === 0` 时传 `is_default = true`
3. 刷新列表

调整后：

1. 调用 `save_model_config`
2. 若为新增模式，拿到返回 id
3. 调用 `set_default_model(id)`
4. 刷新列表

判定新增模式的条件：

- `editingModelId` 为空

### 7.2 SettingsView 编辑模型流程

编辑模式保持：

1. 调用 `save_model_config`
2. 保留原 `is_default`
3. 刷新列表

不追加 `set_default_model`

### 7.3 手动设为默认

在模型列表中新增按钮：

1. 当 `m.is_default === false` 时显示 `设为默认`
2. 点击后调用 `set_default_model(m.id)`
3. 成功后刷新列表

## 8. 测试设计

### 8.1 前端测试

至少覆盖以下场景：

1. 新增首个模型后显示为默认
2. 新增第二个模型后，第二个显示默认，第一个取消默认
3. 编辑默认模型后，默认状态不变
4. 编辑非默认模型后，默认状态不变
5. 点击 `设为默认` 后切换成功

### 8.2 后端测试

至少覆盖以下场景：

1. `set_default_model` 会清除其他非搜索模型默认标记
2. `set_default_model` 不影响搜索 Provider 默认状态
3. 删除默认模型后，会自动补一个默认
4. 删除非默认模型后，不改变当前默认
5. 任意时刻最多只有一个默认模型

## 9. 风险与控制

主要风险：

1. 保存接口返回值变更影响现有调用方
2. 删除默认模型后若不补默认，聊天与路由可能回退到首项推断
3. 若默认切换只在前端处理，可能出现多端状态不一致

控制方式：

1. 统一由后端命令保证默认唯一
2. 保存接口改动后同步更新调用点与测试
3. 增加删除默认模型测试，确保不存在“无默认模型”空档

## 10. 结论

本次最合适的优化方式是：

1. 新增模型时自动切换为默认
2. 编辑模型时不改变默认
3. 新增 `set_default_model` 命令统一处理默认互斥
4. 列表中补齐 `设为默认` 按钮
5. 删除默认模型时自动补一个默认

这样既满足“配置第二个以后默认走第二个”的直接诉求，也让默认模型的生命周期规则完整、可维护、可测试。
