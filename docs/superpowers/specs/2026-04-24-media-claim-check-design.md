# Media Claim-Check Attachment Design

## Context

WorkClaw currently receives browser-file attachments from the React runtime as inline payloads. Text attachments are now truncated before being sent to the model, and image attachments have a per-request total payload cap. This protects the most common 413 cases, but large image payloads are still persisted inline in chat `contentParts`, replayed through transcript reconstruction, and passed through runtime orchestration as base64 strings.

OpenClaw's useful pattern here is not a separate vision model call. Its native vision path still gives image bytes to the selected model. The pattern worth adopting is claim-checking: replace oversized inbound media with an opaque `media://inbound/<id>` reference, persist the bytes in a controlled media store, and hydrate the bytes only at the last boundary that actually needs them.

## Goals

- Keep existing user-visible upload behavior: users can attach supported images, text documents, PDFs, and binary documents from the current composer.
- Reduce database, journal, and transcript bloat for large images by storing large binary image payloads outside `contentParts`.
- Preserve native vision routing and provider behavior by hydrating `media://` references before model requests are built.
- Keep old sessions compatible: existing inline image `data` parts must continue to render and route.
- Add focused tests that prove large images are offloaded, model requests still receive usable images, and unsafe media refs are rejected.

## Non-Goals

- Do not change the frontend upload transport yet. The browser may still send inline data URLs to Tauri in this phase.
- Do not add a full media-understanding fallback pipeline for images, audio, or video.
- Do not introduce provider file-upload APIs. Hydrated native vision requests will still use provider-compatible image blocks.
- Do not remove the existing total image budget. Claim-checking controls local persistence and replay size; budget checks still protect outbound provider payloads.

## Architecture

Add a small Tauri-side media store under the runtime root, likely `runtime_root/cache/chat-media/inbound/`. The store should expose three narrow operations:

- `save_inbound_media(bytes, mime_type, original_name) -> SavedMedia`
- `resolve_inbound_media_ref(media_ref) -> MediaPath`
- `read_inbound_media_ref(media_ref, max_bytes) -> MediaBytes`

The media ref format is `media://inbound/<id>`. IDs must be generated by the store and must reject path separators, null bytes, empty IDs, and traversal operators on read. Stored filenames should include a generated identifier and a safe extension derived from the MIME type or original filename.

The first implementation should offload only images above a fixed threshold, for example 2 MB decoded bytes. Smaller images remain inline. Text and PDF handling stays as-is because text is already truncated and PDFs currently need local extraction.

## Data Flow

1. React sends attachments as it does today.
2. `send_message` calls the async attachment normalization path.
3. Image validation still checks per-image and total decoded bytes.
4. During normalization, large image `sourcePayload` values are decoded and saved to the media store.
5. The persisted user message `contentParts` stores an image part with metadata plus `mediaRef`, not inline `data`.
6. Runtime transcript construction sees image parts. Inline `data` parts are used directly; `mediaRef` parts are hydrated from disk and converted into Anthropic/OpenAI-compatible image blocks.
7. UI history rendering supports both old inline `data` image parts and new `mediaRef` image parts. For `mediaRef` without a preview URL, show a lightweight attachment card instead of a broken image.

## Compatibility

Existing inline image parts remain valid. The image content part shape becomes additive:

```json
{
  "type": "image",
  "name": "screen.png",
  "mimeType": "image/png",
  "size": 3145728,
  "mediaRef": "media://inbound/<id>"
}
```

Old shape remains supported:

```json
{
  "type": "image",
  "name": "screen.png",
  "mimeType": "image/png",
  "size": 1024,
  "data": "data:image/png;base64,..."
}
```

If a `mediaRef` cannot be resolved when preparing a model request, fail the send with a clear attachment-specific error. Do not silently drop images from native vision turns, because that would make the model answer about missing visual context.

## Error Handling

- Invalid data URLs or base64 payloads fail as user input errors before writing media.
- Media store write failures fail the send and best-effort delete any files written during that normalization attempt.
- Unsafe or missing `media://` refs fail model preparation with a clear error.
- Total image budget still applies using decoded size before offload and after hydration.
- Store reads must not follow symlinks or paths outside the media store directory.

## Cleanup

The first implementation may use `cache/chat-media/inbound` and rely on the existing desktop cache cleanup command to remove media. A follow-up can add TTL sweeping if large media usage becomes common. This keeps the first patch small and avoids background lifecycle work.

## Testing

- Rust unit/integration tests for media-store ID safety, write/read, MIME/extension preservation, symlink/path traversal rejection, and best-effort cleanup on later validation failure.
- Attachment platform tests proving a large image is persisted as `mediaRef`, not inline `data`, while small images remain inline.
- Runtime transcript tests proving `mediaRef` images hydrate into Anthropic and OpenAI image blocks.
- Frontend rendering tests proving persisted `mediaRef` image parts display as an attachment card rather than a broken `<img>`.
- Regression tests proving existing inline image sessions still reconstruct and send.

## Implementation Plan

1. Add a focused `chat_media_store` module in `apps/runtime/src-tauri/src/commands/`.
2. Add a runtime media directory to `RuntimePaths` or derive it from `cache_dir` in the media-store module.
3. Extend image normalization to offload decoded images above the threshold and return `mediaRef` image parts.
4. Add an async transcript-building path that can hydrate `mediaRef` image parts before model request construction.
5. Keep persisted history reconstruction compatible with inline image parts and new `mediaRef` parts.
6. Update frontend image history rendering for `mediaRef` parts.
7. Run targeted Rust and Vitest coverage, then `pnpm test:rust-fast`.
