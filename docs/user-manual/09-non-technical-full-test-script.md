# WorkClaw 非技术用户全功能远程测试脚本

## 1. 测试目的

本测试用于让一位非技术背景用户尽量覆盖 WorkClaw 当前主要功能入口和关键流程，重点观察：

- 是否容易找到入口
- 是否容易理解每个模块是做什么的
- 是否能够独立完成主要任务
- 是否在关键节点产生犹豫、误解或放弃

本测试不是技术验收，不要求测试者判断功能实现是否正确，也不要求理解模型、技能、MCP、路由等技术细节。

## 2. 测试方式

- 测试方式：远程独立完成
- 建议时长：60 到 90 分钟
- 建议要求：全程录屏，并边操作边说出自己的想法

请提前告知测试者：

- 不需要懂技术
- 不需要担心操作失误
- 遇到不会用、看不懂、找不到入口时，按真实反应记录即可
- 如果某一题卡住超过 2 分钟，可先记录，再继续下一题

## 3. 每题统一反馈格式

请在每个任务后填写下面内容：

- 是否完成：是 / 否
- 难度评分：1-5
- 最卡的地方：
- 最看不懂的按钮或文案：
- 你原本以为它会怎么工作：
- 你还敢不敢继续往下用：敢 / 不敢
- 原因：

## 4. 测试任务

### 任务 1：第一次打开后，判断从哪里开始

目标：先不要乱点，先观察这个产品是做什么的、你会从哪里开始。

完成标准：你能说出这个产品大概是做什么的，以及你会先点哪个入口。

### 任务 2：完成首次基础设置

目标：请把产品设置到“你觉得已经可以正式使用”的状态。

完成标准：你认为已经完成最基本的配置，可以开始真正使用。

观察重点：

- 你是否能自己找到设置入口
- 你是否知道哪些是必须配置的，哪些不是
- 你是否能理解模型、搜索、默认语言、权限模式这些设置大概是什么意思

### 任务 3：发起第一次任务

目标：从首页或你认为合适的位置，发起一个新任务。

建议任务内容：

`请帮我整理一个文件夹里的文件，并先告诉我你打算怎么整理。`

完成标准：你成功发出任务，并知道系统已经开始处理。

### 任务 4：完成一次本地任务闭环

目标：完成一次和本地文件有关的完整任务。

建议任务内容：

- 整理文件夹
- 汇总某些本地资料
- 读取文件后生成总结

完成标准：你拿到了结果，并知道结果在哪里、是否算完成。

### 任务 5：完成一次联网搜索任务

目标：让它帮你查找公开信息并整理。

建议任务内容：

`请帮我搜索某个主题的最新公开信息，并整理成要点，标注来源。`

完成标准：你成功发起联网任务，并能理解结果中的来源信息。

### 任务 6：理解任务执行过程

目标：在任务执行过程中，观察你能不能看懂它正在做什么。

完成标准：你能说出：

- 它现在是否在工作
- 大概在做哪一步
- 你是否放心继续等它完成

### 任务 7：查看和继续历史任务

目标：找到刚才做过的任务，再次打开并继续。

完成标准：你能自己找到历史记录，分辨不同任务，并继续其中一个。

### 任务 8：搜索、删除或导出历史会话

目标：管理你之前做过的任务记录。

完成标准：至少完成下面任意两项：

- 搜索某个历史任务
- 删除一个不需要的历史任务
- 导出一个任务记录

### 任务 9：进入“专家技能”并理解它是做什么的

目标：进入“专家技能”区域，尝试理解这里的用途。

完成标准：你能说出下面三者大概有什么区别：

- 我的技能
- 技能库
- 找技能

### 任务 10：尝试安装或使用一个技能

目标：在“专家技能”中尝试安装一个技能，或者选中一个技能发起任务。

完成标准：你成功使用一个技能开始任务，或者清楚知道自己卡在哪一步。

### 任务 11：理解“创建”和“技能打包”

目标：在不需要真正懂技术的前提下，判断“创建”和“技能打包”分别是给谁用的。

完成标准：你能用自己的话说出：

- “创建”是做什么的
- “技能打包”是做什么的
- 你会不会使用这两个功能，为什么

### 任务 12：进入“智能体员工”并理解它的作用

目标：进入“智能体员工”页面，判断这个模块在做什么。

完成标准：你能说出它和普通“开始任务”有什么区别。

### 任务 13：尝试让一个员工执行任务

目标：选择一个员工或与员工相关的入口，尝试发起一次任务。

完成标准：你知道这个任务现在是“你自己直接发起”还是“交给某个员工处理”。

### 任务 14：尝试使用团队入口

目标：如果你看到团队入口，请尝试把一个稍复杂的任务交给团队。

建议任务内容：

`帮我把一个复杂需求拆解成步骤，并分别执行，再给我总结结果。`

完成标准：你能看懂这是“团队协作”而不是普通单任务模式。

### 任务 15：观察团队协作过程

目标：在团队任务执行时，观察你能否理解阶段、分工、等待、结果汇总这些信息。

完成标准：你能说出：

- 现在是不是多人或多角色在协作
- 你能不能看懂谁在做什么
- 你是否觉得过程可信

### 任务 16：故意给一个模糊或错误的指令

目标：故意发起一个不够清楚的任务，看看你会怎么修正。

完成标准：你能自己找到修正方式，而不是只能放弃。

### 任务 17：遇到风险提示时做判断

目标：如果过程中出现确认、风险、权限相关提示，请按你的真实想法决定是否继续。

完成标准：你能说出：

- 你是否看懂了提示
- 你为什么敢点或不敢点

### 任务 18：中断后重新进入

目标：在某个任务做到一半时离开，再回来继续。

完成标准：你能判断之前内容有没有保留，以及怎么继续。

### 任务 19：找帮助路径

目标：假设你已经不会用了，请找出你会去哪里求助。

完成标准：你能明确指出一个你认为最像“帮助、说明或补救”的地方。

### 任务 20：总体评价

请最后回答下面问题：

- 这个产品最像帮你做什么的？
- 你最容易上手的是哪一部分？
- 你最看不懂的是哪一部分？
- 哪个功能你最不敢用？
- 如果推荐给朋友，你最担心什么？
- 如果只能改 3 个地方，你会改什么？

## 5. 给测试组织者的提醒

如果只有 1 位非技术用户，这份脚本能覆盖的是“可发现、可理解、可走通”，不能替代下面几类专项验收：

- 技术正确性验证
- 技能创建质量验证
- 技能打包和分发验证
- MCP 服务器配置验证
- 飞书、企微、路由等集成验证
- 极端异常场景和稳定性验证

因此，测试结果应被解读为“普通用户易用性审查”，而不是“全量质量验收”。
