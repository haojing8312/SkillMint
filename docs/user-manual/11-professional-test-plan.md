# WorkClaw 专业测试方案

## 1. 文档目的

本文档用于从专业测试视角，对 WorkClaw 当前版本进行系统化验证，目标是形成一套可复用的发布前测试方案。

与非技术用户测试不同，本方案关注的不只是“能不能用”，还包括：

- 功能完整性
- 主流程稳定性
- 异常处理质量
- 风险控制可用性
- 多模块之间的协同一致性
- 回归测试覆盖

本文档适用于：

- 产品发布前验收
- 版本回归测试
- 关键功能改版后的专项验证
- 外部演示或试点前的稳定性检查

## 2. 测试目标

本轮测试目标定义如下：

1. 验证普通任务主流程可稳定完成。
2. 验证设置、任务、历史、专家技能、智能体员工、团队协作等主要模块可独立使用。
3. 验证关键异常场景下，系统有明确反馈、不会让用户失控。
4. 验证高风险操作存在可理解的确认机制。
5. 验证主要模块之间的入口关系、状态衔接、结果留存一致。
6. 为发布决策提供结构化结论，而不是零散体验反馈。

## 3. 测试范围

### 3.1 本次纳入范围

- 首次进入与基础设置
- 模型配置
- 搜索配置
- 默认语言与沉浸式翻译相关设置
- 权限模式与风险确认
- 开始任务
- 单任务执行闭环
- 本地文件相关任务
- 联网搜索相关任务
- 会话历史管理
- 会话导出与删除
- 专家技能
- 技能安装
- 技能启动任务
- 智能体员工
- 员工任务发起
- 团队协作入口
- 团队任务执行过程
- 中断恢复与历史回访

### 3.2 本次不作为主结论依据的范围

以下内容可测试，但不建议把其结果与“普通用户可用性”混在一起：

- 技能创建质量
- 技能打包质量
- 加密分发正确性
- MCP 服务器细节配置
- 飞书、企微等外部集成联调
- 模型厂商兼容性深测
- 极限性能和长时间压力稳定性

这些能力更适合单列专项测试。

## 4. 测试角色分工

建议至少分成以下角色：

### 4.1 主测

负责严格按用例执行、记录结果、复现问题。

### 4.2 观察员

负责记录操作路径、犹豫点、误解点、异常表现、恢复方式。

### 4.3 产品或设计代表

负责判断问题是否属于认知、文案、导航、反馈或流程设计缺陷。

### 4.4 开发支持

负责在出现阻塞时定位日志、确认是否为已知问题、协助判断严重级别。

## 5. 测试环境要求

测试前应统一以下条件：

- 指定测试版本号
- 指定操作系统版本
- 指定是否为干净安装
- 准备至少一个可用模型配置
- 准备至少一个可用搜索配置
- 准备一个可操作的默认工作目录
- 准备若干本地测试文件
- 准备至少一个可安装技能样本
- 准备至少一个员工或团队样本

建议同时准备：

- 干净环境
- 已有历史数据环境

以分别覆盖首次使用与存量使用场景。

## 6. 测试方法

本方案采用四层测试法：

### 6.1 冒烟测试

确认关键入口是否可打开，主流程是否能从头走通。

### 6.2 功能测试

逐模块验证输入、操作、结果、状态、边界条件。

### 6.3 异常测试

故意制造错误输入、中断、返回、取消、删除、模糊任务、权限确认等场景。

### 6.4 回归测试

在版本变更后，重复关键用例，确认旧能力未被破坏。

## 7. 测试通过标准

建议按下面标准进行发布判断：

- `阻断`：主流程无法开始、无法完成、结果丢失、严重误导、频繁崩溃
- `严重`：功能可用但路径中断、状态错误、结果不可判断、风险提示失效
- `中等`：可完成但明显费解、容易误操作、反馈不清晰、历史管理混乱
- `轻微`：文案、样式、提示语、顺序不佳，但不影响完成

建议发布门槛：

- 阻断问题为 0
- 严重问题为 0
- 中等问题有明确处置结论
- 冒烟与主流程用例通过率达到 100%

## 8. 核心测试矩阵

| 模块 | 测试目标 | 关键判断 |
|------|---------|---------|
| 首次设置 | 能否完成可用配置 | 用户是否知道何时算配置完成 |
| 开始任务 | 能否顺利发起任务 | 入口是否清晰，提交后是否有反馈 |
| 单任务执行 | 能否完成闭环 | 是否看得懂过程和结果 |
| 本地文件任务 | 文件相关能力是否可用 | 是否能理解输入、处理和输出 |
| 联网搜索任务 | 搜索与来源展示是否可靠 | 是否知道信息来自哪里 |
| 历史管理 | 是否能找回、删除、导出 | 是否容易定位旧任务 |
| 专家技能 | 是否能理解并使用技能 | 安装、选择、启动是否顺畅 |
| 智能体员工 | 是否能理解员工模式 | 是否分得清普通任务与员工任务 |
| 团队协作 | 是否能理解团队模式 | 是否看得懂分工、阶段与结果 |
| 风险确认 | 是否能防止误操作 | 提示是否清晰、决策是否安心 |
| 中断恢复 | 是否能恢复任务上下文 | 返回后是否知道怎么继续 |

## 9. 详细测试项

### 9.1 首次上手与设置

验证点：

- 是否能找到设置入口
- 是否能完成模型新增或测试
- 是否能完成搜索配置
- 是否能理解哪些是必填，哪些是选填
- 是否能识别默认语言、翻译、权限模式等设置的大致用途
- 保存后是否有明确成功反馈

重点问题：

- 首次使用是否存在“我还没配完，但不知道还差什么”
- 配置完成后，是否清楚下一步该去哪里

### 9.2 开始任务与任务闭环

验证点：

- 是否能从首页直接输入任务
- 是否能提交自然语言任务
- 是否能看到处理中反馈
- 是否能判断任务是在思考、执行还是完成
- 是否能找到最终结果
- 是否能继续追问或补充说明

重点问题：

- 用户是否知道什么时候需要等待
- 用户是否能区分“正在做”和“已经完成”

### 9.3 本地文件能力

验证点：

- 是否能基于本地文件发起任务
- 是否能理解工作目录或文件范围
- 是否能看到文件处理结果
- 修改、整理、汇总类任务是否有稳定输出

重点问题：

- 是否会让用户担心“改坏文件”
- 文件输出位置是否清晰

### 9.4 联网搜索能力

验证点：

- 搜索配置完成后能否真正执行联网任务
- 搜索结果是否包含来源
- 用户能否分辨“模型生成”和“外部来源”

重点问题：

- 如果来源不清楚，用户会显著降低信任

### 9.5 会话历史与结果管理

验证点：

- 是否能看到历史任务
- 是否能搜索历史
- 是否能区分不同任务
- 是否能删除会话
- 是否能导出会话
- 重新打开旧任务后是否知道如何继续

重点问题：

- 历史管理是否只是“能看到”，还是“能管理”

### 9.6 专家技能

验证点：

- 是否理解“我的技能、技能库、找技能”的区别
- 是否能安装技能
- 是否能从技能入口直接启动任务
- 是否能理解“创建”和“技能打包”的适用对象

重点问题：

- 普通用户是否会被创作者能力干扰
- 是否会把“安装技能”和“创建技能”混淆

### 9.7 智能体员工与团队协作

验证点：

- 是否理解员工模式和普通任务模式的区别
- 是否能给单个员工发起任务
- 是否能通过团队入口发起复杂任务
- 是否能看懂团队的阶段、分工、等待和总结

重点问题：

- 用户是否知道什么时候该用员工，什么时候该用团队
- 团队协作过程是否透明可信

### 9.8 风险提示与权限控制

验证点：

- 高风险动作前是否出现提示
- 提示内容是否可理解
- 用户是否知道“继续”和“取消”分别意味着什么
- 风险确认后状态是否一致

重点问题：

- 风险提示过轻，会造成误操作
- 风险提示过重，会导致用户不敢用

### 9.9 中断、返回与恢复

验证点：

- 中途退出后再次进入，历史是否保留
- 用户是否知道从哪里恢复
- 被中断任务是否还能理解当前状态

重点问题：

- 恢复能力差会显著降低专业感和可依赖性

## 10. 建议测试用例分组

### 10.1 P0 冒烟用例

必须全部通过：

1. 打开产品并进入首页
2. 完成基础设置
3. 发起一个普通任务
4. 完成一个本地文件任务
5. 完成一个联网搜索任务
6. 找回历史任务
7. 删除或导出一个历史任务

### 10.2 P1 核心功能用例

建议全部执行：

1. 从专家技能安装或选择技能并发起任务
2. 进入智能体员工并发起员工任务
3. 从团队入口发起复杂任务
4. 观察团队阶段和结果
5. 遇到风险提示并完成决策
6. 中断后恢复任务

### 10.3 P2 扩展用例

根据时间执行：

1. 比较不同入口的认知差异
2. 测试模糊任务纠正
3. 测试多次连续任务管理
4. 测试普通用户对创建、打包等能力的误解程度

## 11. 缺陷记录规范

每个缺陷至少记录：

- 缺陷标题
- 发现时间
- 测试版本
- 所属模块
- 严重级别
- 前置条件
- 操作步骤
- 实际结果
- 预期结果
- 复现概率
- 录屏或截图
- 是否阻断后续测试

建议严重级别划分：

- S0：阻断发布
- S1：严重影响主流程
- S2：影响体验但可绕过
- S3：轻微问题

## 12. 测试输出物

一轮完整测试后，建议至少输出以下材料：

1. 测试执行记录
2. 缺陷清单
3. 主流程通过率统计
4. 模块问题分布统计
5. 发布建议结论

发布建议应至少包含三种结论之一：

- 可以发布
- 修复阻断问题后发布
- 暂不建议发布

## 13. 专业测试与普通用户测试的关系

建议把两类结果分开看：

- 普通用户测试：回答“新手会不会迷路、会不会怕、会不会看不懂”
- 专业测试：回答“产品是否达到版本验收标准”

如果两类结果冲突，优先拆解为两个问题：

- 是设计问题
- 还是实现问题

不要把“用户看不懂”和“功能不可用”混成一个结论。

## 14. 推荐执行顺序

建议执行顺序如下：

1. 干净环境跑 P0 冒烟
2. 已配置环境跑 P1 核心功能
3. 执行异常和恢复测试
4. 记录阻断与严重问题
5. 修复后执行回归
6. 输出发布建议
